# RL-Factory Search-R1 复现踩坑实录

适用场景：在单机 `8 x RTX 4090 24GB` 上，复现 `RL-Factory` 的 `Search-R1` 风格训练，采用 `7 卡训练 + 1 卡 RAG`。

本文不是泛泛教程，而是基于本次真实排障、smoke test、短程训练和工具调用观察整理的复盘笔记。目标不是“看起来像教程”，而是把真正踩过的坑、改过的代码、如何判断训练是否有效，都压成一份能复用的 runbook。

## 1. 时间跨度与范围

- 记录时间跨度：`2026-03-24` 至 `2026-03-26`
- 本文当前有效内容主要来自 `2026-03-24` 到 `2026-03-25` 的实际操作与验证
- 主题：`RL-Factory Search-R1` 在 `8 x 4090` 上的复现、RAG 服务、工具调用、训练观测与指标查看

## 2. 本次实验的最终认识

- 这套链路已经完成端到端验证：RAG 可用，MCP `search` 工具可用，1-step smoke test 跑通，7 卡训练可以启动并进入稳定 step。
- 对 24GB 的 4090，原始面向 A100 的配置不能直接照搬，必须把 RAG 和训练显存隔离开。
- 最稳妥的卡分配是：
  - `GPU 0-6`：训练
  - `GPU 7`：RAG
- RAG 最终采用的是 `GPU 编码器 + CPU FAISS`。它比原始多卡 GPU FAISS 慢，但稳定得多。
- 这套 Search-R1 复现不是“在 `<think>` 内部搜索，搜完再答”，而是显式的 `assistant -> tool -> assistant` 消息流。
- 当前配置下，`max_turns=2` 更像“一轮工具调用后收尾”，不是长链 ReAct 式追查。
- 工具调用日志最终不应该只依赖 MCP 子进程侧记录，更稳的是直接在 RAG `/retrieve` 入口记录。
- 训练是否有效，主要看 reward、validation reward、KL、entropy、response length，不要只看 `seconds/it`。

## 4. 最开始遇到的关键问题

### 4.1 `curl` 看起来打了 RAG，实际上没有

最开始不是 RAG 服务没起来，而是测试请求本身有问题：

- URL 写成了 `http:127.0.0.1:5003/retrieve`，缺少 `//`
- Header 写成了 `-H '{Content-Type: application/json}'`
- 当前 shell 开了代理，`curl` 默认被代理到了本地代理端口，而不是直连 `127.0.0.1:5003`

正确测试方式：

```bash
curl -sS --noproxy '*' http://127.0.0.1:5003/retrieve \
  -H 'Content-Type: application/json' \
  -d '{"queries":["who is the 14th president of niger"],"topk":1,"return_scores":true}'
```

### 4.2 开着代理时，MCP `search` 工具也可能失败

训练环境里开着代理变量时，`requests` 默认会继承这些环境变量。原始 `search.py` 在当前机器上可能报：

- `Missing dependencies for SOCKS support`

这不是 RAG 坏了，而是本地请求被代理逻辑污染了。

### 4.3 原始 RAG 会把 8 张卡都吃掉

原始实现会把 FAISS 索引拷到所有可见 GPU 上，因此同一个 RAG 进程会：

- 占用 8 张 4090
- 每张卡大约占 `5GB-6GB`

这对训练来说基本不可接受，因为 rollout vLLM 初始化时会直接因为显存不足失败。

### 4.4 原始训练失败点不是 reward，也不是工具，而是 rollout 显存

第一次正式训练失败的核心报错来自 vLLM：

- `No available memory for the cache blocks`

这说明当时失败点在 rollout 初始化，而不是训练本身或 reward 逻辑。

### 4.5 `nvcc` 不在 PATH 会阻断 RAG 启动

你在重启 RAG 时遇到过：

```bash
rag_server/launch.sh: line 18: nvcc: command not found
```

这本身不是核心问题，但会把整个启动链路挡住。

## 5. 为什么原始 RAG 会占满 8 张卡

原因是两点叠加：

1. 启动脚本没有限制 `CUDA_VISIBLE_DEVICES`
2. 检索实现默认把 FAISS 索引搬到所有可见 GPU

所以原始行为是：

- 一个 RAG 进程使用所有 GPU
- 索引在多卡间分片
- 每张卡各占一部分显存和 GPU FAISS 资源

这不是 bug，而是“多卡 GPU FAISS”的默认设计。

## 6. 为什么后来这部分显存占用消失了

后来的稳定方案不是“单卡 GPU FAISS”，而是：

- `GPU 7` 放检索编码器
- FAISS 索引放 CPU

之所以没用“单卡 GPU FAISS”，是因为已经实际撞到过一次大显存申请失败，说明单张 4090 根本装不下完整 GPU 索引。

所以最终 tradeoff 很明确：

- 多卡 GPU FAISS：快，但抢训练卡
- 单卡编码器 + CPU FAISS：稳，但检索慢

对 24GB 4090，优先级必须是先跑通。

## 8. 训练参数怎么理解

### 8.1 这几个参数最关键

- `TRAIN_BATCH_SIZE`
- `rollout.n`
- `PPO_MICRO_BATCH_SIZE_PER_GPU`
- `PPO_MINI_BATCH_SIZE`
- `ROLLOUT_GPU_MEMORY_UTILIZATION`
- `ROLLOUT_MAX_NUM_BATCHED_TOKENS`
- `ROLLOUT_MAX_NUM_SEQS`

### 8.2 为什么 7 卡时 `TRAIN_BATCH_SIZE` 优先用 7 的倍数

当前配置里：

- `TRAINER_N_GPUS_PER_NODE=7`
- `rollout.n=2`
- `PPO_MICRO_BATCH_SIZE_PER_GPU=1`

为了满足 batch 相关整除关系，`TRAIN_BATCH_SIZE` 最稳妥就是取 7 的倍数：

- `7`
- `14`
- `21`

因此当前推荐起点是 `14`。

### 8.3 这套 7 卡配置不能和 README 的 A100 基线直接比

你看到 `30-45s/it` 并不说明比 README 的 `8xA100` 更快，核心原因是：

- 我们降了 batch 和 token 预算
- 这是一套专门给 24GB 4090 做的稳妥配置
- README 的时间口径本身也和当前 run 不完全相同

所以这里不能只比 `seconds/it`。

## 10. Search-R1 这套工具调用到底是怎么工作的

这是这次最重要的认识之一。

### 10.1 它不是 `<think>` 内搜索

当前配置里明确是：

- `enable_thinking=False`

因此当前实现不是：

1. 先在 `<think>` 里内部多轮搜索
2. 搜索完成后再开始输出答案

当前实现更像：

1. assistant 生成一段回复
2. 如果回复里有 `<tool_call>...</tool_call>`，系统就执行工具
3. 工具结果作为 `tool` 消息塞回上下文
4. assistant 再继续生成下一段回复

也就是说，它是显式消息流，不是 hidden-thought 工具执行。

### 10.2 一次 rollout 会不会多次调工具

要分两种情况。

#### 情况 A：同一次 assistant 回复里多个工具调用

支持。

因为 parser 会把一条回复里的所有 `<tool_call>` 都提出来，所以一条 assistant 回复里可以并发或批量发多个 query。

#### 情况 B：先查一轮，再根据结果追查第二轮

当前配置下基本不支持。

因为 `max_turns=2` 限制的是 assistant turn 数。当前更像：

1. assistant 第一次回复决定查什么
2. tool 返回
3. assistant 第二次回复收尾

而不是长链式：

1. assistant 查第一轮
2. tool 返回
3. assistant 再查第二轮
4. tool 返回
5. assistant 最终总结

这点和很多人直觉里的“ReAct 代理”不一样。

## 11. 为什么验证样例里经常看不出工具调用

原因不是没调工具，而是验证产物更偏向保存最终输出，而不是中间轨迹。

所以经常会出现：

- `grpo.log` 里看得到 `CallToolRequest`
- RAG 侧能看到真实 `POST /retrieve`
- `validation_samples` 却没有清晰的 tool trace

这也是后来把工具调用审计放到 RAG 入口层的主要原因。

## 12. 工具调用日志最后是怎么处理的

这部分经历了一个演化过程。

### 12.1 最早的方案

最开始想在 MCP 工具侧记录 `search_tool_calls.jsonl`，也就是在 `envs/tools/search.py` 里记录：

- query
- 状态
- 耗时
- 返回摘要

### 12.2 为什么这个方案最后没有作为主方案

因为 Ray worker、MCP stdio 子进程这两层的环境透传并不稳定。即使代码补了几层显式 env 传递，最终对于“真实 query 一定要落盘”这个目标来说，仍然不如直接在 RAG 入口层记录稳。

### 12.3 最终采用的方案

最终采用：

- 直接在 RAG `/retrieve` 入口记录 JSONL

后面看工具调用，优先看：

- `results/<run_dir>/rag_requests.jsonl`

这个文件能直接告诉你：

- 模型查了什么
- 每次查询多慢
- topk 是多少
- 返回的大致内容是什么

## 15. 哪些 TensorBoard 指标最重要

最重要的是：

- `critic/rewards/mean`
- `critic/score/mean`
- `val-core/nq/reward/mean@1`
- `actor/ppo_kl`
- `actor/kl_loss`
- `actor/entropy`
- `actor/grad_norm`
- `response_length/mean`
- `perf/time_per_step`
- `perf/throughput`

### 推荐观察顺序

#### 先看有没有学到东西

- `critic/rewards/mean`
- `val-core/nq/reward/mean@1`

#### 再看有没有训歪

- `actor/ppo_kl`
- `actor/kl_loss`
- `actor/entropy`

#### 再看是不是开始“少查、少答”

- `response_length/mean`

这个指标对 Search-R1 特别重要。如果长度快速掉下去，模型很可能在走“直接答”而不是“检索后答”。

#### 最后看性能

- `perf/time_per_step`
- `perf/throughput`

这些指标主要帮助判断 CPU FAISS 是否拖慢训练。

## 17. 当前这套 Search-R1 复现最值得记住的几点

- 8x4090 上复现 Search-R1，不要让 RAG 跟训练抢同一批卡
- RAG 最稳的是 `GPU 编码器 + CPU FAISS`
- 代理环境会直接影响本地 MCP search 工具，必须显式屏蔽
- 训练日志路径和结果目录必须对齐，否则后面排查很痛苦
- 工具调用不是 `<think>` 内搜索，而是显式 assistant/tool 消息流
- 当前 `max_turns=2` 只偏一轮工具调用，不偏长链追查
- 看训练成效，不要只看 `seconds/it`，一定要同时看：
  - reward
  - validation reward
  - KL
  - entropy
  - response length

## 18. 一句话总结

这次 RL-Factory Search-R1 复现真正落下来的不是一套“完美参数”，而是一套在 8x4090 上能稳定工作、能看清工具调用、能读懂训练指标、也知道哪里会踩坑的工程化认知。后面如果继续把这个仓库做成“复现踩坑实录 / 教学指南”，这篇文档应该作为第一份 Search-R1 主题的基线记录。
